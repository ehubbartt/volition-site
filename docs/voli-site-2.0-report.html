§1  Correctness bug fixes — commit 399730f

Real, user-visible bugs found in the review. Each fix is self-contained.

bug VP refunds could silently lose points

Why: grantPlayerVp (the refund path when a pack open fails) did a plain read-then-write with no optimistic lock, while grantGold had a proper CAS-retry loop. Two concurrent refunds — or a refund racing the bot's own points write — could drop an increment, leaving the member down VP with no trace.
Fix: both VP and GP grants now go through one shared grantColumn helper that re-reads and compare-and-swaps with a few retries.

Files: src/lib/server/playerStats.ts

bug Wallet→GP conversion could settle items but not credit GP

Why: convertWalletToGold flipped wallet items to paid_out=true first, then credited GP. If the GP credit failed, the items were already burned (no longer claimable in-game) but no balance was added — the member permanently lost the value, with only a console.error.
Fix: on a failed credit it now rolls the item-settle back (items stay claimable) and fires a postOpsAlert — the money-losing event is surfaced, not swallowed.

Files: src/lib/server/playerStats.ts

bug A transient stats outage could demote a member on Discord

Why: every external fetch returns null on failure, and a missing source degrades that score component to 0. A momentary WiseOldMan/Temple 429 was indistinguishable from "no data," so checkRank would compute an artificially low rank and write it to players.rank — which the bot mirrors to a Discord role.
Fix: when Temple or WikiSync is unavailable, the breakdown still renders but the rank is not persisted, with a note explaining why. No demotion off a bad read.

Files: src/routes/me/+page.server.ts

bug Unescaped RSN lookups could match the wrong player

Why: several ilike('rsn', rsn) lookups passed the raw RSN, so a name containing _ or % could match the wrong row (or several). A careful escaping helper existed but wasn't used everywhere.
Fix: exported rsnExactPattern and routed every players and vs_users RSN lookup through one shared findPlayerRow helper / the escape function.

Files: src/lib/server/playerStats.ts, users.ts, dinkDrops.ts

cleanup Consolidated the duplicated players read/write logic

Why: the "find by discord_id else rsn" pattern and the optimistic-deduct loop were each copy-pasted ~6× in playerStats.ts.
Fix: one findPlayerRow(discordId, rsn, columns) locator and one casColumn / grantColumn pair now back every read, spend, and grant.

Files: src/lib/server/playerStats.ts

§3  Theming consistency — commit 256bdc0

Mostly mechanical — make everything use the design tokens and the OSRS sprite frames.

theme 50 hardcoded font literals → tokens

Why: 'rsbold'/'rssmall', ui-sans-serif, … was spelled out 50× across 20 files instead of var(--font-heading) / var(--font-body). One place to change the type stack now.

theme Accent & stone literals → tokens

Why: the global layout had six literal #ff981f; the panel-fill colour #4d4030 was scattered. Added a --stone-fill token and routed both through tokens.

theme Flat pages framed; re-rolled buttons use the base button

Why: /dink-check's status banner was a flat gradient card (not sprite-framed), and /me's "Check rank" + the rank-sim buttons hand-rolled a flat outline instead of the bronze OSRS button. Both now match the rest of the interface. The .osrs-badge and the admin role pill were squared off (999px → 3px), and the rank composite bar adopts the new shared .osrs-bar utility.

Files: app.css, +layout.svelte, dink-check, me, admin/rank-sim, admin/admins

§4  Server / data layer — commit be26548

arch Shared fetch helper with a real timeout

Why: Node's global fetch has no default timeout. wom.ts and the Discord-webhook POST in botBridge.ts (which is awaited inside the gamba open action) had none, so a stalled upstream could hang a page load or a form submit indefinitely.
Fix: new src/lib/server/http.ts (serverFetch / getJson / postJson) adds an abort-on-timeout deadline; wom, rankData, and botBridge now route through it.

cleanup Generic fetchAllFiltered<T> & rank-config de-dupe

Why: fetchAllFiltered returned unknown[], forcing a cast at ~15 callers; getRankConfig had no in-flight guard, so a burst of rank checks each hit bot_config.
Fix: made the paginator generic (mirrors selectAll<T>) and added a single-flight guard to the rank-config cache.

arch Cheaper ban check + last type error fixed + docs

Why: the bans-table read ran on every authenticated request including static assets.
Fix: skip it for asset paths (they never render the banned UI). Also fixed the one pre-existing svelte-check error (a Supabase embed cast), and documented the anon-key authorization posture, money-write locking, and error conventions in ARCHITECTURE.md.
Worth knowing: every read and write uses the Supabase anon key with RLS effectively off, so all authorization lives in app code (the isAdmin/isSuperAdmin gates). The generic table editor at /admin/tables/* reads/writes any table on that key — its route guard is security-critical. A future hardening step is a service-role split. (Now noted in ARCHITECTURE.md.)

§5  Rank scoring — commit a39e5de

arch Weights are normalized to sum 1

Why: the composite is Σ(component·weight) and the score→rank thresholds are calibrated for weights that sum to 1. A hand-edited bot_config row whose weights summed to, say, 1.3 would scale every score up and make the thresholds meaningless — silently jumping people's ranks.
Fix: sanitize() now normalizes the weights to sum 1 (keeping their relative emphasis) on both read and write. Also reuses toRankValue() and corrects a stale doc comment.

Files: src/lib/server/rankConfig.ts, admin/rank-sim/+page.server.ts

§6  Dink auto-tracking — commit 09d35c9

The pipeline with the most issues. It isn't live in prod (static config only), so it was safe to fix fully, including a schema change. Requires the migration in the checklist above.

Headline bug: the auto-credit insert omitted the legacy proof_url/proof_path columns that the manual submit path still writes — if those are NOT NULL, every auto-credit was failing its constraint and retrying forever, silently. The insert now mirrors the manual path and the error is logged.
ChangeWhy
Mirror proof_url/proof_path, tag source='dink' Likely-NOT NULL columns were blocking inserts; source lets revert match a real column, not a review_note text prefix.
Log dup-check & insert errors; treat unique-violation as duplicate Errors were swallowed, hiding the bug above and risking infinite retries. The new unique index makes idempotency correct even across instances.
Scope collect-N progress to the matched tile_id Two tiles tracking the same item pooled their progress (matched on item only). Drops now carry their tile.
Mark consumed partials consumed; revert restores them Partials were never consumed, so after a revert a single new drop could instantly re-credit. Now they're rolled in and restored on revert.
Simulator suppresses the public feed An admin simulating a drop against an open event could post a real bingo-credit announcement to Discord.
Clone is snapshot-and-restore safe cloneTemplateToEvent deleted tiles then inserted; a failed re-clone left the board empty. It now restores the prior tiles on failure.
The migration SQL (db/scripts/dink_tracking_hardening.sql) 
alter table vs_dink_drops add column if not exists tile_id text;

alter table vs_bingo_completions add column if not exists source text;

create unique index if not exists vs_bingo_completions_one_approved
    on vs_bingo_completions (event_id, user_id, tile_id)
    where status = 'approved';

Deferred — reviewed but intentionally not done

Lower-value or higher-risk items from the review I left alone, so you know they were considered.

  • Scoring math for clogAvailable — the field is fetched but doesn't affect the score. It's either a latent bug or intended; I didn't change rank math without confirming the bot's original intent.
  • Batching the rank-sim "apply" — it writes N players sequentially. Fine for the current clan size; a bulk upsert is a future optimization.
  • ModalShell extraction — the 7 board/bingo modals each re-roll the backdrop/escape handling. High value but each has bespoke sizing/animation, so it's a careful refactor, not a mechanical one.
  • Full ConfirmDialog adoption — 8 destructive actions still use native confirm(). Mechanical but per-site wiring; do the high-traffic admin pages first.
  • isTileOpenAt consolidation — three copies of the "is this tile open at time T" check remain; worth unifying but touches the manual submit path too.

All of these are described in full in docs/CODE-REVIEW-2.0.md (§5.3, §2.10, §2.11, §6.7).
