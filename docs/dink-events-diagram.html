Dink → Bingo Auto-Tracking

How an in-game drop auto-credits a bingo tile, what's left to do, and how to test it safely.

RuneLite / Dink dink-proxy (Cloudflare Worker) Supabase (DB) volition-site (SvelteKit)

End-to-end flow

🛠️ Admin builds the event setup
/admin/events/[slug]/builder — tiles + per-tile tracked items (item id/name → tile). Stored in vs_event_tracked_items. Set status open.
🎮 RuneLite + Dink
Player gets a drop. Dink fires a LOOT notification to its configured webhook (the proxy).
⚙️ Proxy serves config /config/<token>
Each member pastes their personal config URL (from the Discord /dink command or the site's /dink-check) into Dink's Dynamic Config URL. The token is validated against dink_tokens (∪ VALID_TOKENS) and is revocable/rotatable. The served config also injects the active events' tracked-item names into Dink's loot allowlist, so those items are sent regardless of value (minLootValue stays 3M — no firehose).
⬇POST /hook/<token>/achievements
⚙️ dink-proxy Worker — handleHook
Validates token, parses the LOOT payload, then makes two independent decisions:
Decision A — Discord feed
Forward to the achievements webhook only if total value ≥ FEED_MIN_VALUE. Cheap tracked items are recorded but not posted.
Decision B — match
If playerName ∈ participants (clan ∪ signups) and item ∈ tracked items → write a row to vs_dink_drops (deduped by drop_key, via ctx.waitUntil).
⬇reads manifest (cached ~30s)
🗄️ vs_active_participants
clan RSNs ∪ open-event signups
🗄️ vs_active_tracked_items
tracked items for open events
🗄️ vs_dink_drops
raw matched drops, processed=false
⬇POST /api/dink/process (cron)  +  throttled backstop on board load
🧮 Site consumer — processDinkDrops()
Drains unprocessed drops → resolves rsn → vs_users → matches tile → timing gate: only credits if the tile was open at the drop's received_at (event started + that row released). Else discarded. Idempotent (no double-credit).
⬇
🗄️ vs_bingo_completions approved
one row per credited player+tile
🏁 /bingo/[slug]
tile shows complete + leaderboard updates, live
Key idea: the proxy stays "dumb" (filter + record); all crediting logic lives on the site. Auto-tracking is a no-op until the proxy has Supabase credentials, so it's safe to deploy partway.

How to test (without opening a real event)

Admin → Dink Drop Simulator   /admin/dink-test

  1. Build an event, add tracked items, set its start time in the past, and set status to preview (admin-only, not public).
  2. Dry run — pick the event, an RSN, and a tracked item; it reports whether it would credit and why (match / identity / timing), writing nothing.
  3. Simulate — inserts a real vs_dink_drops row and runs the consumer end-to-end; then open /bingo/<slug> (visible to you as admin) to see the tile credited.

This exercises the whole site pipeline (match → timing → credit) with no need for the proxy or a live in-game drop. To also test the proxy leg, curl a synthetic LOOT payload at /hook/<token>/achievements and confirm a vs_dink_drops row appears.
