Dink → Bingo Auto-Tracking

How an in-game drop auto-credits a bingo tile, what's left to do, and how to test it safely.

RuneLite / Dink dink-proxy (Cloudflare Worker) Supabase (DB) volition-site (SvelteKit)

End-to-end flow

🛠️ Admin builds the event setup
/admin/events/[slug]/builder — tiles + per-tile tracked items (item id/name → tile). Stored in vs_event_tracked_items. Set status open.
🎮 RuneLite + Dink
Player gets a drop / unlocks a clog slot / gets a pet. Dink fires a LOOT or COLLECTION notification to the proxy.
⚙️ Proxy serves config /config/<token>
Each member pastes their personal config URL (from the Discord /dink command or the site's /dink-check) into Dink's Dynamic Config URL. The token is validated against dink_tokens (∪ VALID_TOKENS) and is revocable/rotatable. The served config also injects the active events' loot tracked-item names into Dink's loot allowlist, so those items are sent regardless of value (minLootValue stays 3M — no firehose). Collection unlocks aren't value-gated, so they always arrive.
⬇POST /hook/<token>/{achievements,collection}
⚙️ dink-proxy Worker — handleHook
Validates token, parses the payload, then makes two independent decisions:
Decision A — Discord feed
Forward to the channel webhook as before (LOOT only forwards above FEED_MIN_VALUE). Forwarding is unchanged from the original proxy.
Decision B — match
If playerName ∈ participants (clan ∪ signups) and item ∈ tracked items of the matching type (LOOT→loot tiles, COLLECTION→collection/pet tiles) → write a row to vs_dink_drops with its notif_type & value (deduped by drop_key, via ctx.waitUntil).
⬇reads manifest (cached ~30s)
🗄️ vs_active_participants
clan RSNs ∪ open-event signups
🗄️ vs_active_tracked_items
tracked items (+ match_type) for open events
🗄️ vs_dink_drops
raw matched drops, outcome null until processed
⬇POST /api/dink/process (cron)  +  throttled backstop on board load
🧮 Site consumer — processDinkDrops()
Drains unprocessed drops → matches tile by type → timing gate (tile open at the drop's received_at?) → resolves rsn → vs_users → collect-N (accumulate quantity until required_qty) → credits. Every drop is stamped with an outcome (credited / no_tile / no_user / timing / duplicate / partial). Idempotent (no double-credit).
⬇
🗄️ vs_bingo_completions approved
one row per credited player+tile
🏁 /bingo/[slug]
tile shows complete + leaderboard updates, live
📣 Discord feed
"X auto-completed a tile" → DISCORD_BINGO_WEBHOOK_URL (open events only)
🔎 /admin/dink-drops
every matched drop + its outcome — debug "why no credit?", reprocess or un-credit.
🧪 /admin/dink-test
dry-run / simulate a drop end-to-end; mint the self-test event; revoke tokens.
Key idea: the proxy stays "dumb" (filter + record); all crediting logic lives on the site. Auto-tracking is a no-op until the proxy has Supabase credentials, so it's safe to deploy partway — and with Supabase unset the proxy behaves exactly as the original (serve config, validate VALID_TOKENS, forward hooks, strip @mentions).

What you can build on a tile

💰 Loot tile match_type=loot
Completed by a LOOT drop of the item. Add the item (id/name) to the tile in the builder. Default.
📖 Collection / 🐾 Pet tile match_type=collection
Completed by a collection-log unlock of the item. Pets are clog slots, so a "get the pet" tile uses this with the pet's item id. Needs Dink's collection-log notifier on.
🔢 Collect-N required_qty
Set a quantity > 1 and the tile credits only once the player accumulates that many (e.g. 100 dragon bones). Progress shows as partial in the drops view.
Combat-achievement / level / KC tiles aren't wired yet — we need a captured Dink payload for those notifier types first (do a Dink "Send Test" for each and we can slot them into the same framework).

Recommended test run

A) Real end-to-end (proves the whole chain):
  1. In /admin/dink-test, click Create / refresh self-test event (an always-open board tracking Bones, Cowhide, Feather, Raw chicken, Big bones).
  2. Open /dink-check, copy your personal config URL into Dink's Dynamic Config URL.
  3. Log in and kill a few cows/chickens. Within ~30s the drop appears on /dink-check and the tile completes on /bingo/dink-self-test.
  4. If nothing shows, open /admin/dink-drops → filter Didn't credit to see the reason (identity / timing / no tile), fix it, and Reprocess.
B) No game needed (proves the site pipeline):
  1. Build a preview event with tracked items and a start time in the past.
  2. /admin/dink-test → Dry run (verdict only, writes nothing) → Simulate (inserts a real drop + runs the consumer), then view the board.

Does this break the existing proxy? No.

The proxy's original job — serve /config/<token> and forward /hook/<token>/<channel> to the Discord webhooks (with @mentions stripped) — is unchanged:
  • Token validation is now the union of VALID_TOKENS and active dink_tokens. Existing VALID_TOKENS still validate; if Supabase is unset it falls back to VALID_TOKENS only — identical to before.
  • Forwarding is untouched: LOOT still only forwards above FEED_MIN_VALUE; COLLECTION and every other type still forward exactly as before (the new code only does a background DB write, then falls through to the same forward).
  • All new DB work is best-effort (ctx.waitUntil, errors swallowed) and gated on Supabase being configured, so it can never block or fail a hook/config request.
