The idea in one paragraph

The Dink RuneLite plugin already fires a webhook whenever a player gets a notable drop or unlocks a collection-log slot. Instead of pointing Dink straight at a Discord channel, each member points it at the dink-proxy — a small Cloudflare Worker. The proxy does two things: it serves Dink its config (so we control what gets sent) and it receives the drop webhooks. For each incoming drop it makes two independent decisions — forward it to the Discord feed (as before), and, if the dropper is a clan member and the item is tracked by an open event, record the drop in the database. The website then drains those recorded drops and auto-credits the matching bingo tile. The proxy stays "dumb" (filter + record); all the crediting logic lives on the site next to the rest of the bingo code.

What Dink is

Dink is a popular RuneLite plugin that sends a webhook (HTTP POST with a JSON body) when something notable happens in-game: a valuable LOOT drop, a COLLECTION log unlock, a level-up, a pet, etc. Normally you give Dink a Discord webhook URL and it posts screenshots to a channel. The two settings that matter to us:

Dynamic Config URL
A URL Dink periodically fetches to get its settings. Pointing this at the proxy lets us control Dink centrally (which notifiers are on, the loot value threshold, the loot allowlist) and means the webhook target is our proxy, not a raw Discord URL.
Notifiers
LOOT (drops above a gp value), COLLECTION (clog unlocks — pets are clog slots too), and others. The proxy's served config decides which fire and how.

The proxy's two jobs

① Serve config GET /config/<token>
Returns the Dink settings JSON for that member. It starts from a bundled template (dinkconfig-template.json) — or, if configured, the admin-editable dink_config row in bot_config — substitutes the member's {{TOKEN}}, and injects the tracked-item names of every open event into Dink's loot allowlist. That allowlist is the trick that lets cheap tracked items (e.g. Bones) reach us even though minLootValue stays high — so the feed isn't flooded but tracked items always arrive. Collection unlocks aren't value-gated, so they always come through.
② Handle hooks POST /hook/<token>/<channel>
Dink POSTs each notification here. The proxy validates the token, then makes the two decisions below: forward to Discord and (best-effort, in the background) record a matched drop to vs_dink_drops. It also strips @mentions so member-authored text can't ping the channel.

Validates the token, parses the payload, then makes two independent decisions:
Decision A — Discord feed

Decision B — record a match

Deduped by drop_key, done in the background via ctx.waitUntil so it never blocks the hook.

live view: one row per tile a player is currently on (open-event tiles + personal boards), each typed item/manual. The two views below derive from its item subset.
🗄️ vs_active_participants
distinct owners of any active item tile (lowercased) — derived
🗄️ vs_active_tracked_items

🗄️ vs_dink_drops
raw matched drops; processed=false, outcome null until the site runs. Durable recent-drops log for race healing.

Every drop is stamped with an outcome so admins can see why it did or didn't credit. Idempotent — it can't double-credit.

2.0: matching is now per player against the live vs_active_player_tiles index, not per event — the drop is matched only against the dropper's own active item tiles (open-event tiles and their personal collection-log board), so one drop can credit both an event tile and a board tile. The timing gate is the activation rule: a drop credits only if its received_at is at/after when that tile became active (event start, or the board's creation) — never a drop you already had. The cron path also runs a bounded reconcile that re-checks recent un-credited drops to heal ordering races, still subject to the activation rule.

How a drop is matched

A drop is recorded (by the proxy) when both hold:

  • the dropper's RSN is in vs_active_participants — the clan roster (players) plus anyone signed up to an open event; and
  • the looted item matches a row in vs_active_tracked_items for an open event — by item id (preferred — robust against noted/charged name variants), else by name, honoring the optional source (NPC/container) constraint, and only against tracked items whose match_type equals the notification type.

Then, when the site processes it, the RSN is resolved to a vs_users row (the dropper must have onboarded on the site) and the drop is matched against that user's own active item tiles in vs_active_player_tiles (their open-event tiles and their personal collection-log board), so credit is always per-user-correct. Bingo tiles are binary, so the first qualifying drop credits the tile (collect-N tiles accumulate first — see below).

The site consumer & outcomes

The consumer (src/lib/server/dinkDrops.ts → processDinkDrops()) is the brain. It runs from a cron hitting POST /api/dink/process (shared-secret guarded) and, as a backstop, a throttled call when a bingo board is loaded. Each drop ends with one verdict, stored on the row so /admin/dink-drops can explain it:

OutcomeMeaning
creditedTile credited for this player (an approved completion was inserted).
partialA collect-N tile that isn't complete yet — counted toward the total, waiting for more.
consumedA prior partial that has now been rolled into a completed collect-N tile, so it can't be double-counted. new in 2.0
duplicateThat player already has an approved completion for the tile (or a concurrent run beat this one).
timingThe tile wasn't open at the drop's time (event hadn't started, or the row hadn't released) — discarded, same rule as a manual submission.
no_tileNo tracked item matched (or the event has no tiles / isn't set up).
no_userThe RSN doesn't resolve to a site user — they haven't onboarded, so we can't attribute it.
revertedAn admin un-credited it from the drops view; it stays processed so it won't re-credit.

Tile types you can build

💰 Loot tile match_type=loot
Completed by a LOOT drop of the item. Reaches the proxy via the injected loot allowlist regardless of value. The default.
📖 Collection / 🐾 Pet tile match_type=collection
Completed by a collection-log unlock. Pets are clog slots, so a "get the pet" tile uses this with the pet's item id. Needs Dink's collection-log notifier on.
🔢 Collect-N required_qty > 1
Credits only once the player accumulates that many of the item across drops (e.g. 100 dragon bones). In-progress drops show as partial; when it completes they become consumed.
Combat-achievement / level / KC tiles aren't wired yet — those notifier payloads need to be captured first (do a Dink "Send Test" for each), then they slot into the same framework.

Per-user config URLs (Dink tokens)

Each member uses a personal config URL — ${PROXY_BASE_URL}/config/<token> — pasted into RuneLite → Dink → Advanced → Dynamic Config URL. The token identifies the member and is revocable/rotatable.

Tokens live in dink_tokens (token, discord_id, created_at, revoked_at) and are minted two equivalent ways — both write the same table keyed by Discord id, so a member has one active token however they got it:

Discord bot
/dink (mint / fetch), /dink-revoke (admin), and auto-revoke when a member leaves the clan.
Site
/dink-check — any member can get / copy / rotate their link; admins revoke from /admin/dink-test. (src/lib/server/dinkTokens.ts)

The proxy validates an incoming token against the union of active dink_tokens rows and its legacy VALID_TOKENS secret (cached for TOKENS_TTL_MS), so a site-minted token works with no Cloudflare API call and a revoke/rotate takes effect within the cache TTL.

2.0 hardening & the migration

The auto-tracking pipeline was reviewed and hardened on the voli-site-2.0-refactor branch (it isn't live yet, so this was safe to do). What changed in the consumer:

  • The credit insert was likely failing silently. It omitted the legacy proof_url/proof_path columns the manual submit path writes (likely NOT NULL), so every auto-credit could fail its constraint and retry forever. It now mirrors those columns and tags the row source='dink'; errors are logged instead of swallowed.
  • Collect-N progress is now scoped to the specific tile (via a new vs_dink_drops.tile_id), so two tiles tracking the same item no longer pool progress, and completed partials are marked consumed so a post-revert drop can't instantly re-credit.
  • A unique index on approved (event, user, tile) makes idempotency correct even under concurrent / multi-instance processing — a race now resolves to duplicate instead of a double credit.
  • The admin simulator suppresses the public feed, and revertDinkCredit matches by the new source column instead of the human-readable note text.
One DB step before going live: run db/scripts/dink_tracking_hardening.sql in the Supabase SQL editor. It adds vs_dink_drops.tile_id, vs_bingo_completions.source, and the unique index. (The unique index assumes no duplicate approved completions already exist — true for a fresh Dink dataset.)

Unified active-tiles index (this branch). Matching moved from per-event tracked items to the single live vs_active_player_tiles view, and the Dink consumer now also credits personal collection-log boards. The activation rule (no credit for a drop received before the tile was active) is enforced in the consumer, and a bounded reconcile pass heals ordering races. Apply db/scripts/active_tiles.sql and db/scripts/personal_boards.sql in the SQL editor. The proxy's vs_active_participants/vs_active_tracked_items views are recreated as derived from the index, with the same column shapes, so the proxy needs no change.

How to test it

A) No game needed (proves the site pipeline):
  1. Build a preview event with tracked items and a start time in the past.
  2. /admin/dink-test → Dry run (verdict only, writes nothing) → Simulate (inserts a real drop + runs the consumer; the public feed is suppressed), then view the board.
B) Real end-to-end (proves the whole chain):
  1. In /admin/dink-test, create the always-open self-test event (tracks Bones, Cowhide, Feather, Raw chicken, Big bones).
  2. Open /dink-check, copy your personal config URL into Dink's Dynamic Config URL.
  3. Kill a few cows/chickens. Within ~30s (manifest cache) + the processor interval the drop appears on /dink-check and the tile completes on /bingo/dink-self-test.
  4. Nothing showing? /admin/dink-drops → filter Didn't credit for the reason (identity / timing / no tile), fix it, and Reprocess.

Does this break the existing proxy? No

The proxy's original job — serve /config/<token> and forward /hook/<token>/<channel> to the Discord webhooks with @mentions stripped — is unchanged:
  • Token validation is the union of VALID_TOKENS and active dink_tokens. If Supabase is unset it falls back to VALID_TOKENS only — identical to before.
  • Forwarding is untouched: LOOT still only forwards at/above FEED_MIN_VALUE; everything else forwards exactly as before. The new code only does a background DB write, then falls through to the same forward.
  • All new DB work is best-effort (ctx.waitUntil, errors swallowed) and gated on Supabase being configured, so it can never block or fail a hook/config request.
